# Replay & Reprocessing (full rebuild v1)

> One button: **completely clean and rebuild the KB.** Throw away everything the
> pipeline derived — candidates, entities, claims, outputs, reviews, caches — keep every
> immutable Source, reset all pipeline status, and let the machine flow through and
> re-derive the whole KB from scratch. The first concrete build of the Replay stage
> (LIFE-10/12), scoped to **full rebuild only**; partial/selective replay is deferred.

## 1. Intent (the why / JTBD)

SPEC-0004 Stage 8 promises that *"knowledge can be rebuilt as understanding improves"* —
as models, prompts, and stage logic evolve, the KB should be able to **reprocess preserved
Sources into improved derived knowledge without losing the ground truth** (LIFE-10). Today
we have a pipeline that derives candidates (SPEC-0015/0020), entities (SPEC-0020), claims
(SPEC-0016), and reviews (SPEC-0018) from immutable Sources (SPEC-0008/0014) — but **no way
to redo it**. If a stage prompt improves, or a bad run leaves the derived layer in a state
you don't trust, the only recourse is hand-deletion or starting a new vault.

The job the Principal hires this for: *"I changed/fixed something upstream — wipe what the
machine inferred and rebuild it cleanly from my preserved Sources, without me losing a
single Source."* The trigger is a deliberate, occasional decision ("this got better, redo
all of it"); the payoff is a clean derived layer regenerated by the current pipeline, with
the ground truth and full lineage intact.

A second, deliberate use: **Replay doubles as the Principal's pipeline test/validation tool.**
Because it re-derives the *entire* KB from Sources through the **normal, unmodified pipeline**,
running it on demand exercises every stage end-to-end — a fast way to confirm the machine still
produces a coherent KB after a change, and to reproduce the derived layer from a known-clean
state. This is why the rebuild MUST run the live pipeline verbatim and in capture order, not a
replay-only path (REPLAY-14).

This spec intentionally **starts simple**: the full-rebuild mode of LIFE-12 (*"discard the
entire existing derived KB and rebuild from preserved Sources"*), delivered as a single
confirmed action in Settings. It is the foundation the later partial/selective replay
(LIFE-12/13) and replay-with-new-secondaries (LIFE-11) build on — but those are out of
scope here.

Why it's safe to be destructive: the only thing destroyed is **derived** knowledge, which
is *by definition* reproducible from Sources (DATA-1: Sources immutable, Entities/Outputs
derived). Sources are never touched; the purge itself is a git commit, so even the
discarded derived generation remains recoverable in history (DATA-11).

### 1.1 Where derived knowledge lives now (CANON / STAGING)

Since SPEC-0019 (CANON) and SPEC-0021 (STAGING), the vault has **two long-lived branches**
and derived knowledge spans both:

- **Evergreen `main`** — the published view (`sources/` + resolved derived: `entities/`
  written *only* by resolution/CONNECT per CANON-5, plus `claims/`, `outputs/`). The
  **promotion gate is the sole writer of `main`** (STAGING-3).
- **Working zone on `staging`** — where every stage works (STAGING-2) and where the
  pre-resolution / machinery state lives: `candidates/`, the `reviews/` queue, `queue/`,
  audit indexes/caches. `main` never contains these working paths by construction
  (STAGING-6); the working zone is durable + versioned, i.e. replayable (CANON-11).

A full replay therefore has to clear derived state in **both** zones and let the promotion
gate republish an evergreen `main` for the rebuilt generation — without ever exposing a
half-purged `main` (CANON-1/3).

## 2. Scope

**In scope:**
- A **Full Replay** action in the **Settings** view (SPEC-0017 SHELL), behind a
  **confirmation dialog** naming the consequence.
- **Purging all derived knowledge** in both zones: `candidates/`, `entities/`, `claims/`,
  `outputs/`, the open `reviews/` queue, and rebuildable caches/indexes — everything below
  the Source line.
- **Preserving all Sources** (primary + secondary): `source.md`, raw bytes, and the
  archival lineage in each source's `audit.jsonl`.
- **Resetting all pipeline status** so every Source is reprocessed from the first stage,
  including items previously **set-aside** (permanently failed) or **awaiting-review**.
- **Preserving the inbox**: un-archived captures are left in place and flow through the
  rebuilt pipeline normally.
- **Auto-resuming** the pipeline so the rebuild "just happens" after confirmation, and the
  **promotion gate** republishing `main` to the rebuilt evergreen set.

**Out of scope (for now):**
- **Partial / selective replay** — "reingest these X" / replay a time-window or a single
  Source (LIFE-12 partial mode, LIFE-13 merge-and-review semantics). Full rebuild only.
- **Producing new secondary sources during replay** — no re-fetching/re-researching
  (LIFE-11's optional "may produce new secondaries"). Replay re-derives from the existing
  archive only.
- **Named/queryable retention of superseded derived generations** beyond what git history
  already gives (DATA-11). We do not build a "generations" browser here.
- **Type-to-confirm / multi-step guards, scheduling, or dry-run/preview** of what will be
  purged — a single yes/no confirm is enough for v1.
- **Selective stage replay** (e.g. "redo claims only, keep entities") — full pipeline only.
- **Changes to the promotion gate or stage harness themselves** — Replay *uses* the
  SPEC-0021 staging/promotion mechanism; it does not modify it.

## 3. User flows / feature surface

**Primary flow — Full Replay:**
1. The Principal opens **Settings** (SPEC-0017 SHELL).
2. Under a **Replay / Maintenance** section there is a button: **"Clean & Rebuild KB…"**
   with a one-line explanation ("Delete all derived knowledge and reprocess every Source
   from scratch. Your Sources are preserved.").
3. Clicking it opens a **confirmation dialog**: *"Completely clean and rebuild this KB?
   This permanently deletes all derived knowledge — candidates, entities, claims, and
   review questions — and reprocesses every Source from scratch. Your Sources are
   preserved. This cannot be undone from the app."* Buttons: **Cancel** / **Clean &
   Rebuild**.
4. On confirm, the app:
   a. **Pauses** the pipeline (stops the stage sweeps; lets the in-flight item's commit
      finish, see Open Questions).
   b. On the **`staging`** branch, **purges** all derived artifacts and resets all pipeline
      status (see §4), advancing `staging` as committed work.
   c. Runs the **promotion gate** so **`main`** is republished to hold only the (now
      cleared) evergreen set — `main` advances atomically and is never half-purged.
   d. **Resumes** the pipeline, which sweeps the preserved Sources and re-derives the KB
      from the first stage forward (Decompose → candidates → resolution → claims → …).
5. The existing pipeline-status surface reflects the rebuild in progress (queue depth
   climbs back up, then drains) — no new progress UI is required for v1.

**Secondary / edge flows:**
- **Cancel** in the dialog → nothing happens; the pipeline keeps running.
- **Replay already running** → the button is disabled / the action is a no-op until the
  current replay's purge step completes (a second concurrent purge is prevented).
- **Empty KB (no Sources yet)** → replay is still safe: there's nothing derived to purge
  and nothing to reprocess; it completes as a no-op (promotion is idempotent, STAGING-4).
- **Interrupted mid-purge** (app crash/quit) → the operation is restartable; branch state
  is the source of truth (STAGING-8). On restart the reset either completes or `staging`/
  `main` are left at their pre-purge committed state — never half-purged and dirty.
- **Answered reviews that became Sources** are preserved (they are Sources, LIFE-7); only
  the **open `reviews/` queue artifacts** (working-zone decision points) are purged.

### REPLAY-3 — Sources are sacred across replay
- **Status:** draft · **Priority:** must
- **Statement:** Full Replay **MUST NOT** delete or edit any Source — neither `source.md`,
  the verbatim raw content, nor the **archival** events (`captured`, `archived`, and the
  replay-epoch markers of REPLAY-6) in each Source's `audit.jsonl`. Both **primary and
  secondary** Sources are preserved (LIFE-11: both are immutable archive).
- **Rationale:** Ground Truth Is Sacred (PRIN-1). Replay's entire safety argument is that it
  only ever destroys *reproducible* derived knowledge; the moment it can touch a Source it
  stops being safe and stops being "replay."
- **Traces:** DATA-1, DATA-2, LIFE-2, LIFE-10, LIFE-11, PRIN-1
- **Verify:** test:replay (`replay.test.ts` — source.md identical + audit only grows across replay)

### REPLAY-4 — Purge everything below the Source line, in both zones
- **Status:** draft · **Priority:** must
- **Statement:** Full Replay **MUST** remove all derived artifacts wherever they live: the
  evergreen derived trees (`entities/`, `claims/`, `outputs/`) and the working-zone derived
  state on `staging` (`candidates/`, the **open** `reviews/` queue, `queue/`, audit
  indexes, and `.kb/cache/` incl. stage worktrees). It **MUST NOT** purge `.kb/config.json`
  (vault identity) or the inbox (REPLAY-7).
- **Rationale:** Everything below the Source line is derived and reproducible (DATA-1). A
  *clean* rebuild means the new generation is produced solely by the current pipeline from
  Sources, not contaminated by stale prior-generation artifacts — including the
  pre-resolution `candidates/` buffer (CANON-4) and machinery state (CANON-9). Open reviews
  are working-zone decision points tied to a derived generation; they are purged. (Principal
  answer-notes that were promoted to **Sources** under LIFE-7 are Sources and survive.)
- **Traces:** DATA-1, DATA-4, LIFE-12, CANON-9
- **Verify:** test:replay (`replay.test.ts` — candidates purged on staging; entities cleared from main)

### REPLAY-6 — Reset status append-only, via a replay epoch
- **Status:** draft · **Priority:** must
- **Statement:** The stage queues are computed by scanning each Source's `audit.jsonl` for
  terminal stage markers (`decomposed`, `claimed`, `setaside`, `awaiting-review`, …). Full
  Replay **MUST** reset this **without deleting or rewriting** those events. Instead it
  **appends** a `replay-reset` event carrying a monotonically increasing `replayId` (epoch);
  the stage state-readers **MUST** consider only stage markers appended **after** the latest
  `replay-reset` event. Rebuildable caches/indexes (e.g. `.kb/cache/status.json`) are
  reset/rebuilt freely.
- **Rationale:** The audit log is append-only and is the lineage backbone (DATA-10): we must
  not erase the record that a prior generation existed. An **epoch marker** makes "this
  Source is unprocessed again" a forward-only fact, keeps every Source immutable except for
  its own append-only audit, preserves full history for Audit, and keeps the stages
  idempotent/restartable (ORCH-13) — a re-poke after reset simply re-derives. The working
  zone that records this is durable + versioned by design (CANON-11), so the reset replays
  cleanly.
- **Coupling:** this requires the existing stage state-readers (decompose/claims/resolution,
  and any future stage) to filter markers by the latest epoch. That is an ORCH-stage change
  driven by this spec; see Open Questions and the SPEC-0014 changelog when implemented.
- **Traces:** DATA-10, ORCH-4, ORCH-13, CANON-11
- **Verify:** test:replayEpoch (`replayEpoch.test.ts` + `replayReaders.test.ts` — readers honor only post-epoch markers; history preserved)

### REPLAY-8 — Atomic via staging + the promotion gate
- **Status:** draft · **Priority:** must
- **Statement:** The app **MUST** pause the pipeline (stop sweeps; quiesce in-flight work),
  then perform the purge (REPLAY-4) + epoch reset (REPLAY-6) + cache reset **on the
  `staging` branch** as committed work (stages only ever write `staging`, STAGING-2). The
  **promotion gate** (the sole writer of `main`, STAGING-3) then republishes `main` to the
  cleared evergreen set. The evergreen **`main`** tree **MUST NOT** be left half-purged or
  dirty for the user or Obsidian to observe — it advances only through the gate (CANON-1/3).
- **Rationale:** A half-deleted derived layer is worse than either state, and a half-purged
  `main` would violate the evergreen invariant (CANON-1). Reusing the existing staging →
  promotion discipline gives atomicity, crash-safety, and "the messy middle never appears in
  `main`" for free, and keeps replay consistent with how every other stage mutates the vault.
- **Traces:** CANON-1, CANON-3, CANON-8, STAGING-2, STAGING-3, STAGING-4, ORCH-3
- **Verify:** test:replay (`replay.test.ts` — staging committed cleanly; main clean + entities cleared via the gate)

## 5. Open questions

- [x] **Epoch-filter placement** — RESOLVED (impl): the filter lives in **one shared helper**,
      `kb/replayEpoch.ts` `epochScopedLines(raw)`, used by all stage state-readers
      (`readDecomposeState`, `readClaimsState`, `readConnectState`) via a one-line change each —
      no per-stage duplication. Recorded as the ORCH coupling in the SPEC-0014 changelog.
- [x] **In-flight item at pause** — RESOLVED (impl): `pipeline.fullReplay()` calls `stop()` on
      every stage (orch/decompose/connect/claims) before the purge, then `runFullReplay` takes
      the shared canonical-writer lock — so any in-flight commit drains (sub-second ff) before
      the purge runs, and no new sweep starts. The epoch reset un-marks the item so it
      reprocesses fresh; stages are restarted afterward (auto-resume, REPLAY-9).
- [ ] **Replay audit destination** — REPLAY-11 wants a vault-level record of the replay, but
      the global audit index is deferred (SPEC-0014 open q). v1 likely colocates per-Source
      `replay-reset` markers (already required) plus a single top-level `replay` event in the
      working zone; promote to the global audit index when that exists.
- [ ] **Outputs not yet built** — `outputs/` is reserved in the vault layout but no stage
      writes it yet. REPLAY-4 lists it for completeness; harmless to purge a missing tree.
- [ ] **Empty `entities/` is expected post-reset** — until resolution/CONNECT runs,
      evergreen `entities/` is honestly empty (CANON-10); the v1 rebuild populates
      `candidates/` first, then `entities/` once CONNECT lands. Replay does not special-case
      this — it just resets and lets the pipeline re-derive.
- [ ] **Future: partial replay** — this spec's epoch mechanism (REPLAY-6) is designed to
      generalize: a *partial* replay would append `replay-reset` to a **subset** of Sources.
      Confirm the marker shape is forward-compatible with LIFE-12/13 partial mode.
